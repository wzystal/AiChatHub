# LightChat 项目分析与技术方案

## 1. 项目定位

LightChat 是一个 Android AI 流式聊天 Demo，核心目标不是做成完整聊天产品，而是验证两件事：

1. AI 回复的流式输出如何在 Android 端稳定接入。
2. 如何在消息持续增量更新时，只刷新最后一条消息，避免列表全量重绘带来的卡顿。

这个项目适合面试时重点讲三件事：

- SSE / chunk 流式处理
- RecyclerView 局部刷新优化
- MVVM + StateFlow 的单向数据流设计

## 3. 功能模块拆解

### 3.1 聊天输入模块

用户在底部输入框输入内容，点击发送后，UI 会把文本交给 ViewModel。

实现点：

- `EditText` 负责输入
- `Button` 负责发送 / 停止生成
- `ChatActivity` 通过 `sendCurrentInput()` 把内容交给 `ChatViewModel`

原理：

- 空字符串会被直接过滤
- 生成中禁止重复发送，避免并发请求覆盖当前流式输出

### 3.2 消息列表模块

聊天记录用 `RecyclerView` 展示，每条消息是一个独立 item。

实现点：

- `ChatAdapter` 维护本地消息列表
- `append()` 使用 `notifyItemRangeInserted()`
- `updateAssistant()` 使用 `notifyItemChanged(position, payload)`
- `finishAssistant()` 只更新最后一条 assistant 消息的完成态

原理：

- 新增 user/assistant 消息时，只插入新 item
- token 增量到来时，只刷新最后一个 assistant item
- 这样可以避免 `submitList()` 或全量 `notifyDataSetChanged()` 带来的抖动

### 3.3 流式输出模块

当前项目默认使用 Mock 流式客户端模拟 token 逐段返回；同时保留了可接真实 OpenAI 兼容接口的 SSE 客户端。

实现点：

- [MockAiStreamClient.kt](/Users/zhaoyang.wzy/work/codex/aichatdemo/app/src/main/java/com/example/aichatdemo/network/MockAiStreamClient.kt) 通过 `flow { delay(); emit(Token) }` 模拟流式数据
- [OkHttpSseClient.kt](/Users/zhaoyang.wzy/work/codex/aichatdemo/app/src/main/java/com/example/aichatdemo/network/OkHttpSseClient.kt) 通过 OkHttp 读取 `text/event-stream`

原理：

- SSE 的数据会以 `data:` 行的形式持续返回
- OpenAI 兼容接口通常会把增量内容放在 `choices[0].delta.content`
- 客户端逐行解析即可拿到 token

### 3.4 状态管理模块

ViewModel 同时维护两类状态：

- `StateFlow<ChatUiState>`：保存完整聊天状态
- `SharedFlow<ChatUiEvent>`：发给 UI 的增量刷新事件

对应文件：

- [ChatUiState.kt](/Users/zhaoyang.wzy/work/codex/aichatdemo/app/src/main/java/com/example/aichatdemo/ui/chat/ChatUiState.kt)
- [ChatUiEvent.kt](/Users/zhaoyang.wzy/work/codex/aichatdemo/app/src/main/java/com/example/aichatdemo/ui/chat/ChatUiEvent.kt)
- [ChatViewModel.kt](/Users/zhaoyang.wzy/work/codex/aichatdemo/app/src/main/java/com/example/aichatdemo/ui/chat/ChatViewModel.kt)

原理：

- `StateFlow` 用来描述“当前完整页面状态”
- `SharedFlow` 用来描述“一次 UI 增量事件”
- 这样可以把“状态”与“列表局部更新动作”分开，避免把所有 UI 逻辑都塞进同一个状态对象里

## 4. 核心技术方案

### 4.1 MVVM 单向数据流

用户输入后，数据流向是：

`UI -> ViewModel -> Repository -> Stream Client -> ViewModel -> UI`

这个链路的好处是：

- UI 不直接依赖网络细节
- Repository 不关心页面怎么画
- ViewModel 统一管理消息追加、流式更新、结束态和错误态

面试时可以这样说：

> 我把业务状态集中在 ViewModel，UI 只负责渲染和派发事件，这样流式数据来了以后不会把业务逻辑散落到 Activity 或 Adapter 里。

### 4.2 流式数据节流

在 [ChatViewModel.kt](/Users/zhaoyang.wzy/work/codex/aichatdemo/app/src/main/java/com/example/aichatdemo/ui/chat/ChatViewModel.kt) 中，token 并不是每来一个就刷新一次 UI，而是做了时间窗口控制：

- `throttleMs = 75L`
- 累积 token 到 `StringBuilder`
- 到达时间窗口后再统一刷新最后一个 assistant item

这样做的意义：

- 降低高频重组/重绘次数
- 避免 token 很碎时 UI 卡顿
- 让列表滚动更稳定

### 4.3 RecyclerView payload 局部刷新

这是项目里最重要的性能点。

实现位置：

- [ChatAdapter.kt](/Users/zhaoyang.wzy/work/codex/aichatdemo/app/src/main/java/com/example/aichatdemo/ui/chat/ChatAdapter.kt)

关键做法：

- 新消息使用 `notifyItemRangeInserted`
- assistant token 更新使用 `notifyItemChanged(index, Payload.Content(content))`
- 完成态切换使用 `notifyItemChanged(index, Payload.Finished(content))`

原理：

`payload` 会让 RecyclerView 走局部绑定逻辑，而不是整个 ViewHolder 重建。
这意味着只有文本内容变化，气泡样式和布局结构不会被反复重算。

面试表达建议：

> 我没有用 `submitList()` 去做流式阶段的每次更新，因为那会触发 Diff 计算和更多绑定开销。对于“最后一条消息一直变”的场景，payload 局部刷新更直接，也更符合性能目标。

### 4.4 流式阶段纯文本，完成后再切换完成态

文档里要求“流式阶段纯文本，结束后再做 Markdown 渲染”，当前实现已经保留了这个思路，只是 UI 端还没有引入 Markdown 库。

现在的策略是：

- 流式过程中直接用纯文本拼接
- 结束时把 `isStreaming` 置为 `false`
- 后续如果接 Markdown 渲染，可以只在完成态开启 Markdown 解析

这样做的原因：

- 流式期间每个 token 都做 Markdown 解析成本较高
- 流式文本可能不完整，提前渲染容易出现格式抖动
- 完成后一次性渲染更稳定

## 6. 网络层设计

### 6.1 Mock 流式实现

文件： [MockAiStreamClient.kt](/Users/zhaoyang.wzy/work/codex/aichatdemo/app/src/main/java/com/example/aichatdemo/network/MockAiStreamClient.kt)

当前默认实现是模拟流式输出，适合演示和本地联调。

实现原理：

- 根据最后一条用户消息拼一个模拟回答
- 用 `chunked(2)` 将内容拆成小 token
- 每隔 35ms 发出一个 `Token`
- 最后发 `Done`

这能很好地模拟真实大模型“逐段返回”的体验。

### 6.2 真实 SSE 实现

文件： [OkHttpSseClient.kt](/Users/zhaoyang.wzy/work/codex/aichatdemo/app/src/main/java/com/example/aichatdemo/network/OkHttpSseClient.kt)

实现思路：

- 使用 OkHttp 发起带 `Accept: text/event-stream` 的请求
- 从 response body 的 source 中逐行读取
- 过滤 `data:` 开头的行
- 遇到 `[DONE]` 标记结束
- 对 OpenAI 兼容格式解析 `choices[0].delta.content`

为什么这样设计：

- OkHttp 可靠、成熟，适合 Android 端接流式接口
- 逐行读取能兼容 SSE 规范
- 与 UI 无关，可以替换成别的模型服务而不用改页面代码

## 7. UI 实现细节

### 7.1 页面组成

文件： [ChatActivity.kt](/Users/zhaoyang.wzy/work/codex/aichatdemo/app/src/main/java/com/example/aichatdemo/ui/chat/ChatActivity.kt)

页面分三段：

- 顶部状态栏文本
- 中部消息列表
- 底部输入区

### 7.2 自动滚动

每次收到 UI 事件后，都会滚动到最后一条消息：

- `recyclerView.scrollToPosition(maxOf(adapter.itemCount - 1, 0))`

这个体验细节很重要，因为流式输出天然是“最后一条在变”，用户大概率会关心底部最新内容。

### 7.3 停止生成

当前项目已经支持“停止生成”按钮：

- 点击后取消当前流式 job
- 将 assistant 消息标记为非 streaming

这在面试里可以作为一个“可扩展能力”来讲，说明你对长请求中断和 UI 状态收口有考虑。

## 8. 性能优化总结

项目最核心的性能优化点有四个：

1. 只更新最后一条 assistant 消息。
2. 使用 `notifyItemChanged(position, payload)`，避免整表刷新。
3. 流式 token 做 75ms 节流。
4. 流式阶段不做 Markdown 解析，降低每次更新成本。

这几个点共同保证了：

- UI 不容易卡顿
- 列表不闪烁
- 长文本场景更稳
- 流式体验接近真实产品

## 9. 面试时怎么讲这个项目

你可以按这个顺序讲：

1. 项目目标：验证 Android 端 AI 流式输出和列表局部刷新。
2. 架构设计：MVVM 分层，UI 和网络解耦。
3. 核心难点：token 级更新不能全量刷新。
4. 解决方案：StateFlow + SharedFlow + RecyclerView payload + throttle。
5. 网络扩展：Mock 流和真实 SSE 都保留，便于演示和接生产接口。
6. 性能收尾：流式阶段纯文本，完成后再切 Markdown。

一句话总结可以这样说：

> 这是一个基于 Kotlin + MVVM 的 AI Chat Android Demo，支持 SSE 风格的流式输出，并通过 RecyclerView payload 局部刷新与 token 节流机制，实现了消息列表的高频增量更新优化。
